# Vault Hygiene

The vault accumulates knowledge across months and branches. Without hygiene it becomes a graveyard of stale plans and half-written notes. With hygiene it stays useful.

## Rules

1. **One ADR per decision.** Decisions live in `decisions/`, not commit messages. If a colleague asks "why did we pick X over Y?", the answer should be a vault link, not a story.
2. **Bugs get post-mortems only when non-obvious.** Trivial typo fixes do not need a `bugs/` note. Anything that took >1 hour to diagnose, recurred, or could recur — write it up.
3. **Plans have lifecycle.** A plan is born in `plans/planning/`, moves to `plans/active/` when work starts, and ends in `plans/legacy/completed/` (shipped) or `plans/legacy/superseded/` (replaced). Update the `status:` frontmatter at every move.
4. **Guides accumulate gradually.** Do not seed every guide up-front. Write a guide when you have solved the same problem twice.
5. **No secrets, no NDA.** The vault is your knowledge base, not a credential store. If a note would be problematic if pushed to a public repo, it should not be in the vault.
6. **Wikilinks over filepaths.** `[[2026-04-28]]` survives moves; `daily/2026-04-28.md` breaks. Always prefer wikilinks for cross-references.

## Daily Notes

Daily notes are append-only logs. Each session adds:
- A row to the Sessions table (time, persona, task, files, outcome)
- Optional decision-trail blocks for in-the-moment choices
- Links to bugs, plans, decisions touched during the session

Do not rewrite history in daily notes. If something was wrong, add a follow-up entry — keep the original.

## Weekly Notes

Generated by `/weekly`. Aggregate without paraphrasing: include direct links to source dailies so the reader can dig in. The weekly is an index, not a rewrite.

## Templates

All notes are created from `templates/`. To change note structure, edit the template once. Existing notes do not get retro-fitted automatically — that is intentional.

## When to Move Plans to Legacy

- **Completed:** All success criteria in the plan are met, the feature shipped. Move on the same day; do not let `active/` accumulate dead plans.
- **Superseded:** A better approach was adopted. Add `superseded_by: [[new-plan]]` to the original's frontmatter. Add `supersedes: [[old-plan]]` to the new one.

## When to Prune

- **Daily notes:** Never prune. They are cheap and the historical record is valuable.
- **Plans in `legacy/`:** Never prune unless the project domain changes entirely. Old plans inform new decisions.
- **Empty feature folders:** Prune when a feature is fully retired. Keep the parent feature folder even if all its plans moved to `legacy/`.

## Anti-Patterns

- **Mega-decision notes.** One ADR documenting 10 decisions = none of them are findable. Split them.
- **Plans that never close.** A plan in `active/` for 6 months is a sign of either an abandoned project or a missing lifecycle update.
- **Vault as scratch pad.** Use `memory/` for personal notes if needed, but the vault is not where you draft tweets or grocery lists.
- **Duplicate guides per project.** If you maintain multiple projects, share one vault with shared `guides/` and per-project subdirectories for plans/decisions.
